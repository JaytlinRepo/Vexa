# Knowledge Feed Implementation Guide

## Overview

The Knowledge Feed is a curated insights system that automatically extracts learnings, patterns, trends, and content angles from feed data, brand memory, and performance metrics. It helps CEOs (users) understand what's working and what patterns are emerging across their content and audience.

## Architecture

### Database Schema

#### Enums

**KnowledgeType** - Categorizes the type of knowledge:
- `insight` - High-level observation or discovery
- `pattern` - Recurring trend or behavior
- `learning` - Lesson learned from feedback or performance
- `trend_summary` - Summary of trending topics
- `content_angle` - Potential content hook or angle
- `audience_signal` - Signal about audience preferences
- `competitive_advantage` - Unique advantage discovered

**KnowledgeSource** - Tracks where knowledge originated:
- `feed_item` - From curated feed (articles, videos, trends)
- `brand_memory` - From accumulated user feedback/preferences
- `trend_report` - From Maya's trend reports
- `user_feedback` - Direct user input during meetings
- `performance_metric` - From post performance data
- `ai_analysis` - Generated by AI analysis

#### Knowledge Model

```typescript
model Knowledge {
  id              String           @id @default(uuid())
  companyId       String           @map("company_id")
  type            KnowledgeType
  source          KnowledgeSource
  title           String           // Clear, actionable title
  summary         String           // Rich summary with context
  details         Json?            // Extended data, recommendations, analysis
  tags            String[]         // Searchable keywords
  sourceUrl       String?          // Link to original source
  relatedItemIds  String[]         // Links to related knowledge items
  relevanceScore  Float            // 0-1 relevance to this company
  isArchived      Boolean          // Keep learnings fresh
  createdAt       DateTime
  updatedAt       DateTime
}
```

### Backend Services

#### KnowledgeService (`/apps/api/src/services/knowledge.service.ts`)

Core service for knowledge management:

**Main Methods:**
- `createKnowledge()` - Create new knowledge item
- `getKnowledge()` - Fetch with filters (type, source, tags)
- `getKnowledgeById()` - Get specific item
- `updateKnowledge()` - Update item
- `deleteKnowledge()` - Delete item
- `addRelatedItem()` - Link related items
- `extractInsightsFromFeed()` - Create knowledge from feed items (score > 75)
- `extractPatternsFromMemory()` - Create knowledge from brand memory
- `archiveOldKnowledge()` - Soft-delete old items (default: 90 days)
- `searchKnowledge()` - Full-text search

#### API Routes

**Main Knowledge Routes** (`/api/knowledge/*`)

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/` | List knowledge items with filters |
| POST | `/` | Create new knowledge item |
| GET | `/:id` | Get specific knowledge item |
| PATCH | `/:id` | Update knowledge item |
| DELETE | `/:id` | Delete knowledge item |
| POST | `/search` | Search knowledge by keyword |
| POST | `/:id/related/:relatedId` | Link related items |
| POST | `/extract/feed` | Extract insights from feed items |
| POST | `/extract/memory` | Extract patterns from brand memory |
| POST | `/archive-old` | Archive knowledge older than N days |

**Admin Routes** (`/api/knowledge-admin/*`)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/seed-from-memory` | Populate knowledge from existing memory |
| POST | `/archive-old` | Admin cleanup endpoint |
| GET | `/stats` | Knowledge statistics by type/source |

### Frontend Implementation

#### Frontend Data Wire (`/apps/web/public/knowledge-data-wire.js`)

- Fetches knowledge from `/api/knowledge?limit=100`
- Falls back to `/api/feed` if knowledge is empty
- Renders knowledge items with:
  - Type-based icon styling
  - Summary text
  - Tags/categories
  - Time ago indicator
  - Relevance score (0-100)
  - Related items linking

#### HTML Template

The Knowledge view (`#view-db-knowledge`) in `/apps/web/src/app/_prototype/body.html` includes:

- **Masthead** - Shows stats (items today, active sources, save rate)
- **Navigation sidebar** - Feed filters and source list
- **Feed display** - Organized by date with action buttons
- **Context area** - Related trending content

## Usage Flows

### 1. Automatic Knowledge Extraction

**From Feed Items:**
```typescript
const feedItems = await fetchFeedItems(companyId)
const knowledgeItems = await knowledgeService.extractInsightsFromFeed(
  companyId,
  feedItems.filter(item => item.score > 75)
)
```

**From Brand Memory:**
```typescript
const patterns = await knowledgeService.extractPatternsFromMemory(companyId)
```

### 2. User Accessing Knowledge Feed

1. User navigates to "Knowledge" tab
2. `/api/knowledge?limit=100` is fetched
3. Items rendered grouped by date
4. User can filter by type, source, or search

### 3. Linking Knowledge to Tasks

In the future, knowledge items can be:
- Linked to content briefs
- Used by Alex (copywriter) for angles
- Referenced in Jordan's (strategist) plans
- Tracked in brand memory updates

## Integration Points

### 1. Brand Memory ↔ Knowledge

Brand memory is the raw feedback/learning. Knowledge is the refined, actionable extraction.

```
BrandMemory (raw) → KnowledgeService.extractPatternsFromMemory() → Knowledge (curated)
```

### 2. Feed Items ↔ Knowledge

High-signal feed items become knowledge insights.

```
FeedItem (score > 75) → KnowledgeService.extractInsightsFromFeed() → Knowledge
```

### 3. Meetings ↔ Knowledge

User decisions in meetings → brand memory → knowledge extraction

```
Meeting Decision → BrandMemory → Knowledge
```

## Configuration & Tuning

### Relevance Scoring

- Feed items: `Math.min(1, score / 100)` (if score 90, relevance = 0.9)
- Memory items: Use `memory.weight` (0-3 scale)
- Default threshold: 0.5 (50%)

### Archival Policy

- **Default**: Archive items older than 90 days
- **Use case**: Keep learning fresh, avoid stale insights
- **Admin endpoint**: POST `/api/knowledge-admin/archive-old?daysOld=30`

### High-Signal Feed Items

- **Score threshold**: > 75 (on 0-100 scale)
- **Prevents**: Low-quality items cluttering knowledge
- **Rationale**: Only meaningful signals become knowledge

## Performance Considerations

### Indexing Strategy

```sql
CREATE INDEX knowledge_company_id_idx ON knowledge(company_id);
CREATE INDEX knowledge_type_idx ON knowledge(type);
CREATE INDEX knowledge_source_idx ON knowledge(source);
CREATE INDEX knowledge_created_at_idx ON knowledge(created_at);
CREATE INDEX knowledge_is_archived_idx ON knowledge(is_archived);
```

### Query Optimization

- **List endpoint**: Filters by company + archived status, sorts by date
- **Search endpoint**: Full-text search on title/summary/tags
- **Archive endpoint**: Bulk update with date cutoff

### Pagination

- Default limit: 50 items
- Max limit: 100 items
- Offset-based pagination supported

## Support & Debugging

### Common Issues

**No knowledge items showing:**
- Check if brand memory exists (use `/api/knowledge-admin/stats`)
- Run seeding endpoint: POST `/api/knowledge-admin/seed-from-memory`
- Verify relevance score filtering

**Search not working:**
- Ensure tags are lowercase in database
- Check PostgreSQL full-text search configuration
- Try exact title match first

**Performance issues:**
- Check indexes are created
- Monitor query execution time
- Consider archiving old items more frequently

### Debugging Queries

```sql
-- Check knowledge items for a company
SELECT * FROM knowledge WHERE company_id = 'xxx' ORDER BY created_at DESC;

-- Check by type distribution
SELECT type, COUNT(*) FROM knowledge WHERE company_id = 'xxx' GROUP BY type;

-- Check archived items
SELECT COUNT(*) FROM knowledge WHERE company_id = 'xxx' AND is_archived = true;
```
